# 个人主页公开版改版设计

> 日期：2026-07-02
> 范围：`apps/web/src/pages/Home.tsx`（及配套文案）
> 目标受众：**同行与合作者**（非招聘方）
> 整体气质：**Quiet Authority / 留白即权威**（参考 Apple 产品页、Arc Browser 官网、Rauno Freiberg 作品集）

---

## 1. 背景与动机

当前主页 `Home.tsx` 作为对外公开的个人名片，存在三个具体问题：

1. **公司信息直接暴露**：核心优势区显式列出"东华医为、数字马力、泛联新安"三家公司名，履历可被反向追溯。
2. **医疗场景过重**：项目标题"AI + 超声放射信息系统"、影响力文案"服务 12 家三甲医院"、项目区小标题"从医疗 AI 到企业 Agent 平台"——多处主动突出医疗经历，与"AI Agent 工程师"主定位不匹配。
3. **核心优势区是简历腔**："全周期最高绩效"、"BU AI 专项负责人"、"效率与团队建设"——典型的履历标签，三张卡片读完仍是"他做过什么"，不是"他怎么看问题"。

## 2. 设计目标

- **隐藏公司信息**：全文不出现任何公司名、项目代号、行业垂直场景名（医疗、金融、电商等）。
- **弱化医疗经历**：医疗项目不作为独立条目，仅作为"跨领域落地能力"的隐性证据存在。
- **强化 AI Agent 工程师定位**：让访客 30 秒内建立"这是个能交付生产级 Agent 系统的工程师"的判断。
- **核心优势从"履历标签"升级为"工程观点"**：三段文字必须能识别出独立思考，不能被替换成任何同行的内容。

## 3. 视觉气质

**Quiet Authority / 留白即权威**

- 黑白为主，单一 accent（沿用现有 indigo `#4f46e5`）。
- 字号差异拉大：核心优势区使用 `display-xl` 量级，强调对比。
- 减少装饰：去掉图标卡片、tag pill 集群等"开发者个人主页"模板痕迹。
- 3D 知识图谱保留（技术栈区）——这是当前页面少数有记忆点的元素，且与"留白"反差构成节奏。

不引入新组件；不修改设计 token；不修改 `globals.css`、`tailwind.config.ts`。

## 4. 结构与文案（4 段）

### 4.1 Hero（极简立场式）

```
[eyebrow]  AI Agent Engineer · Backend Architect

[h1]       我设计能在生产环境跑起来的智能体系统。

[p]        在 RAG、Agent 协议、LLM 推理与企业级后端架构的交界处，
           把 Demo 变成可被运维的工程。

[pills]    沿用现有 expertise 数组（RAG 知识工程 · MCP/A2A 协议 · 多智能体编排 · 企业级 LLM 应用 · Agent 架构设计）

[CTA]      阅读博客 / 邮件联系
```

**关键变化**：
- 标题从"我是谁"改为"我做什么 + 我的标准"——立场先于身份。
- 副标出现"Demo → 生产"工程张力，同行能立即共鸣。
- 删除原"5 年经验、医疗、企业研发、Serverless"枚举式开场。

### 4.2 能力矩阵（替代"代表项目"）

小标题：**能力域**（视觉上沿用原结构的小标题位置，但内容按"我解决的问题类型"重构）

5 个能力域，按"我解决的问题类型"组织。每域结构：
- 编号 01-05（左对齐、小号）
- 一句话定性（带工程观点）
- 1-2 条证据（具体技术决策或量级数字）
- **不允许出现**：公司名、行业垂直场景名、客户名、医院名

文案：

```
01  RAG & 企业知识工程
    让知识库不再只回答"是什么"，而是回答"在我们的业务里怎么做"。
    · 设计过支持万级文档、6 个业务域、毫秒级召回的检索管线
    · 把 LlamaIndex / Haystack / 自研 Router 拼成可灰度的链路

02  多智能体编排
    Agent 不是 Prompt 拼接，是角色、工具、记忆、终止条件的工程设计。
    · 落地 ReAct / Plan-Execute / Multi-Agent Debate 多种范式
    · 把外部业务封装成 MCP Tools，让 Agent 可观测、可重试

03  LLM 推理与工程化
    关注模型能力上限，更关注推理成本、可控性和团队使用规范。
    · Qwen / vLLM / Claude Code 在企业环境的落地路径
    · 主导团队 AI 编程工具引入与 Code Review 机制

04  企业级后端架构
    不相信"AI 项目"可以脱离传统软件工程。Spring Boot / JVM 是基本盘。
    · 高并发网关、消息中间件、图数据库在 ToB 业务的真实承载
    · 从单体到微服务、从代码到 CI/CD 的全周期治理

05  跨领域落地能力
    在 ToB、ToB-Internal、垂直行业信息化三类场景都做过从 0 到 1。
    让我能快速判断"这个需求是工程问题还是产品问题"。
```

**视觉**：单栏，每域 64-80px 垂直间距，左侧 01-05 编号用 mono 字体 + 淡灰色，字号约 12px。整体无卡片、无图标、无背景色。

### 4.3 核心优势（工程观点型）

标题：**核心优势** · 我对工程问题的三个长期判断。

3 段 pull-quote + 论证。每段结构：
- 大号编号 01/02/03
- 引文（被强调的判断，display-md 量级）
- 论证散文（3-4 行）

```
01  ─────────────────────────────────────
    "真正难的从来不是让 Agent 跑起来，
     是让它在生产里不胡说、不超时、不被人绕过。"

    我把每个 Agent 项目都当成一个分布式系统来设计：
    工具调用要有超时和降级、上下文要有边界、
    失败要有可重入路径。在 6 个业务域的实践中，
    这套工程纪律比任何 Prompt 技巧都更可靠。

02  ─────────────────────────────────────
    "我倾向于做'减法架构'——
     用 3 个组件能解决的事，绝不用 5 个。"

    LangGraph / CrewAI / Dify / Coze 我都用过，
    最后的判断标准始终是：能否在三个月后，
    团队里第二个人能看懂、能改、能上线。
    复杂度必须可被继承，不是炫技。

03  ─────────────────────────────────────
    "技术决策的终点不是'能不能做'，是'值不值得维护'。"

    主导过公司级研发平台接入 200+ 应用、
    内部技术分享 20+ 场。
    我相信工程师的影响力 = 你写的代码 × 别人因为你能少写的代码。
```

**关键约束**：
- 不出现公司名、绩效评级、内部术语（"BU"、"Leader"等）。
- "6 个业务域"、"200+ 应用"、"20+ 场"作为可信度锚点保留，但不透露来源。
- 标题文案避免"我做了什么"，必须落到"我怎么想"。

### 4.4 关于区改写

原 About 三段描述含"5 年后端开发与 AI 工程经验"、"主导设计了企业内部 MCP 工具调用体系、Serverless 智能管控平台、AI 辅助医疗信息系统等多个项目"——后半句直接点名项目和场景。

**新文案**（2-3 句）：

```
你好，我是刘酝泽（Jensen）。
我设计过从 RAG 检索到 Agent 编排的完整工程链路，
也在 ToB、平台型、垂直行业信息化三类场景里做过从 0 到 1 的项目。
关注 Agent 设计模式、协议标准、推理优化；
更关注这些技术能不能被一个团队持续维护。
```

## 6. 数据结构（仅前端，无后端变更）

```ts
// 能力矩阵
interface Capability {
  no: string               // "01" - "05"
  title: string            // 域标题
  stance: string           // 一句话观点（带判断）
  evidence: string[]       // 1-2 条证据
}

const capabilities: Capability[] = [/* 5 项 */]

// 核心优势
interface Principle {
  no: string               // "01" - "03"
  quote: string            // 引文
  argument: string         // 论证散文
}

const principles: Principle[] = [/* 3 项 */]
```

这两个数组替换原文件中的 `projects` 和 `advantages`。
