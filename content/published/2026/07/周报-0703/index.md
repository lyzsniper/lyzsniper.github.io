### 一、主要工作

本周持续推进 **fh-openagent 从 Python(Sanic) 到 Java(Spring Boot)** 的技术栈迁移重构，重点完成 SDK 适配联调、流式架构搭建及 SSE 事件对齐。

### 二、难点与问题

**1. SDK 开发与适配**

AI 辅助迁移生成的 Java SDK（17 个子服务 + 50+ DTO）存在细节缺失，部分交互行为与 Python 原版不一致：
- `PromptEvent.toolArgs` 字段类型差异：Python 是字典，Java 是字符串，导致下游解析行为不同，需手动抓取真实数据流样本逐一校准
- 错误模型映射不完整：Python 的 9 种子类异常与 Java 异常体系的语义对应需逐一手动核实，部分边界场景（如连接超时、重试策略）尚未对齐

**2. Python 与 Java 运行时适配**

两种语言的网络模型差异大，使得 opencode 沙箱调用和流式数据响应的适配难度超出预期：
- 同步请求走 Apache HttpClient5，流式走 Reactor Netty，两套客户端需要各自测通，排查线程安全与超时配置
- SSE 流式响应需同时处理两条通道——主通道的 `streamPrompt` 流 + 旁路事件订阅（提问/回复/驳回/TODO 更新），Java 的 Reactor 线程模型与 Python asyncio 差异较大，旁路事件合并逻辑需要重写
- SSE keepalive 的心跳、done 信号发送、异常中断的资源释放逻辑需要重新设计状态机

**3. 构建与环境问题**

- 主项目与子模块 Gradle 版本不一致（父模块 v5.3 vs 子模块 v8.5），导致 JDK 17 下编译报错，需定向指定 wrapper 路径
- SpringDoc 版本与 Spring 6.2 兼容性冲突，Swagger 接口文档缺失，已修复
- 测试阶段出现 19 个 `ClassNotFoundException`，排查后发现是测试模块依赖声明遗漏

**4. UI 渲染能力丢失**

Python 版已有的 AUIP 卡片渲染（机票搜索结果卡片、证件表单卡片等）在迁移到 Java 后视觉效果和生成逻辑丢失，需要重新实现卡片组装器与前端 SSE 协议的对接。

**5. 旧 Python 代码遗留的质量问题**

迁移过程中发现 Python 原始项目存在若干严重问题，Java 版需要修复而非照搬：
- 1095 行死代码（已废弃的路由文件未被注册）
- 日志中明文记录 token 和完整 HTTP 响应体（合规风险）
- SSE 流式异常路径可能重复发送 done 信号，导致前端重连风暴
- 中文路由正则无边界，可被错误输入绕过

**6. 日志体系统一**

Python 版使用 structlog + 日志文件，Java 版需统一接入公司日志平台：
- 日志落 Redis（与 AI 项目共用 DB 15 但物理隔离不同 list）
- 敏感字段脱敏规则需与 Python 版对齐（apiKey、token、Authorization 等）
- 请求链路追踪需通过 HTTP 拦截器注入 reqSeqNo

**7. 灰度部署策略**

Python 和 Java 两套 Hub 需要并行运行一段时间，共用 `work/` 工作目录和 opencode 沙箱网络：
- 灰度期 Java 只读，避免双写冲突
- nginx 按业务场景 category 切流，需定义清晰的双跑验收标准
- 回滚方案需具备，确保切流后出问题能在数分钟内切回 Python
